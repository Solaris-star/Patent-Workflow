# patent-deslop：专利文本去 AI 味

输入一段或一篇专利文本（part 文件、docx 抽取文本、粘贴片段均可），输出改写后的文本 + 改动清单。**只改语言，不改技术事实**——技术特征、数据、专利引用一字不动；改写后术语必须与 facts_ledger.terminology 一致（有 ledger 时）。

## 检测清单（AI 痕迹 → 专利文体）

### 1. 词汇层（硬替换）

| 禁用 | 替换为 |
|---|---|
| 进一步（对外文档腔） | 所述 / 在此基础上 / 具体地 |
| 客户、贵方、用户朋友 | 所述用户 / 操作人员 |
| 首先…其次…最后… | 按流程顺序直接叙述，用「随后」「在…之后」衔接 |
| 值得注意的是、需要指出的是 | 删除，直接说事 |
| 综上所述（正文中段） | 删除或改「由此」 |
| 赋能、助力、打造、亮点 | 用具体技术动词：实现、执行、生成、判定 |

### 2. 句式层

- **排比句**（「第一…第二…其三…」「不仅…而且…更…」连排）→ 拆为连贯段落，信息按逻辑递进而非对仗排列。
- **列表腔**（该用连贯段落的地方用了 bullet/编号列表，尤其背景技术与实施方式）→ 改连贯段落，保留「S1、S2…」步骤编号（这是专利规范，不是 AI 痕迹）。
- **过度工整**（每段字数相近、每段都是「总-分」结构、小标题强迫症）→ 打破节奏，按技术叙述自然展开。
- **解释腔**（「这样做的好处是」「之所以这样设计是因为」）→ 改「由此」「从而」直接接效果。
- **汇报腔**（「本方案完成了」「我们实现了」「测试表明效果良好」）→ 改「本发明实施例提供」「所述模块用于」。

### 3. 结构层

- 空洞总结段（结尾无信息量的「本发明具有广阔前景」类）→ 删除或替换为具体有益效果回指。
- 同义反复（同一效果换三种说法凑字数）→ 合并为一处最准确的表述。
- 悬空指代（「该方法」「上述系统」指代不明）→ 补「所述XX」明确指代。

### 4. 体裁层（专利规范正向要求）

- 技术特征引用用「所述」体：首次出现定义全称，后文「所述XX」。
- 转折用「然而」「同时」「此外」，禁「其缺陷在于」「缺点是」。
- 实施方式叙述用「本发明实施例提供……」，结尾保留法定套话。
- 无实验支撑的具体数字（准确率/提升百分比）→ 标注「示例性数据，需实际测试」或删除。

## 执行流程

1. **检测**：通读文本，按四层清单标记所有命中点（位置 + 类型）。
2. **AI 浓度评级**：high（≥8 处命中或整段排比）/ medium（3-7 处）/ low（≤2 处），报告给用户。
3. **改写**：逐处改写；保持字数在原文 ±20% 内（字数约束由 patent-draft 负责，本 skill 不越权扩写）。
4. **输出**：改写后全文 + 改动清单（每条：位置 → 原文片段 → 改后片段 → 命中类型）。
5. 在 patent-review 回改闭环中被调用时：改动清单回填对应 edit 的 structured_diff 条目。

## 边界

- 不改技术方案内容、不增删技术特征、不动专利号与引用信息。
- 不处理一致性问题（术语漂移、图号错位是 patent-review 一致性审计员的职责，此处只保证不制造新漂移）。
- 网文等非专利文本的去 AI 味请用 story-deslop，本 skill 的目标文体是专利交底书。
