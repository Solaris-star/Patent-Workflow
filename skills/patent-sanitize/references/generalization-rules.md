# 上位化规则表（generalization-rules）

apply 模式的改写细则，单一真源。原则：**保留技术特征，抹去实现指纹**——改写后本领域技术人员仍能理解并复现方案，但无法反推出内部系统、客户与真实运行数据。

## 逐类规则

### 1. 内部系统/模块代号（internal_codename）

- 改写为**功能性上位词**，格式优先「所述 + 功能名 + 模块/单元/引擎」：
  - `Phoenix-Gateway` → `所述接入网关模块`
  - `Odin 调度器` → `所述任务调度模块`
  - `mars-pipeline` → `所述数据处理流水线`
- **全文一致**：同一代号在全文（含附图 .mmd/.drawio 节点文字）只对应一个上位词；上位词登记进 facts_ledger.terminology。
- 内部版本号（`v3.2-hotfix`）删除或改「预设版本」。

### 2. 具体阈值/魔法数

- 判定阈值 → 「预设阈值」；时间窗 → 「预设时长」；容量/配额 → 「预设容量」。
- 若阈值本身是技术特征（权利要求可能依赖取值范围），改为**去业务指纹的范围表述**：「0.87」→「预设置信度阈值（示例性地，可取 0.8-0.9）」——保范围、去精确内部调参值。

### 3. 真实性能指标（real_metric）

- 默认删除整句或改定性表述（「显著降低」「有效减少」）。
- 需要数字支撑有益效果时：保留**量级**并强制标注——「实测 P99 延迟 137ms」→「示例性地，响应延迟可控制在百毫秒级（示例性数据，需实际测试）」。
- 与 patent-draft 数据真实性规则同源：任何无实验支撑的精确数字都不允许存在，无论是否涉密。

### 4. 客户名/合作方（client_name）

- → 「第三方平台」「外部系统」「合作方系统」；涉及行业特征需保留时用行业泛称（「某银行」→「金融机构信息系统」）。

### 5. 基础设施指纹（infra）

- 域名/URL → 删除或「远程服务端」；IP/网段 → 删除；文件路径 → 「本地存储」「配置文件」；仓库/桶/集群名 → 「代码仓库」「对象存储」「计算集群」。
- 交底书正文**本就不该出现**任何 URL/IP/路径——出现即改，无论是否在 map 里。

### 6. 人名（person_name）

- → 删除或「操作人员」「测试人员」。发明人信息不属于正文内容，出现即删。

### 7. 业务/行业指纹（补充，源自 v5 §7.5）

- 内部业务线/产品线名 → 通用业务描述（「XX 曝光竞价链路」→「内容分发排序流程」）。
- 内部分类体系 → 抽象替代（「一类/二类/三类客户」→「第一类别/第二类别/第三类别对象」）。

## 改写质量自检（apply 收尾必做）

1. 用 validate_sanitize 复扫改写产物：map 词条零残留。
2. 上位词全文一致性：同一 entry 的 replacement 无变体。
3. 技术特征完整性：改写前后逐段对照，确认没有把「怎么实现」改丢，只改掉了「在谁家/哪套系统上实现」。
4. 可读性：上位词符合专利文体（「所述」体），不产生新的指代悬空。
