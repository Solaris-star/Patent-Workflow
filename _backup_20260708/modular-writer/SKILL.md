# Modular-Writer - 分块撰写引擎

## 概述

本技能用于长文本的分块撰写和管理，解决长文本生成中的以下问题：
1. **一次性生成长度限制** - LLM 单次输出长度有限
2. **修改成本高** - 修改一处需要重新生成全文
3. **版本管理困难** - 多次修改后无法追溯历史版本
4. **跨部分依赖** - 修改某部分可能需要同步更新其他部分

**核心设计理念：**
- 分块生成 → 每部分独立生成、独立落盘
- 单独修改 → 每部分可独立修改，不影响其他部分
- 联动检测 → 修改后自动检测是否需要同步更新其他部分
- 版本管理 → 每次修改自动保存历史版本
- 合并输出 → 全部完成后合并为最终文档

## 工作模式

### 模式 A：专利模式

**适用场景：** 专利技术交底书撰写

**分块结构（5 部分，2026-03-08 更新）：**
```
1. 技术领域（part_01）
2. 背景技术（part_02）
3. 发明内容（part_03）
   - 3.1 要解决的技术问题
   - 3.2 技术方案
   - 3.3 有益效果
4. 附图说明（part_04）
5. 具体实施方式（part_05）
```

**注意：** 权利要求书由专利代理师根据交底书撰写，发明人无需提供，本技能不生成权利要求书。

**特有功能：**
- IPR 模拟审查（形式审查 + 实质审查）
- 专利术语检查
- 与 deep-research / patent-workflow 联动（背景专利分析）

**分块撰写规则（强制）：**
1. **逐块生成** - 每次只生成一个部分，不得一次性生成多个部分
2. **逐块确认（默认）** - 默认每部分生成后请求用户确认再继续；若用户明确授权“连续生成/一次性初稿（例如连续生成 part_01-05）”，则允许不中断推进，最后统一 review（在共享频道注意控制刷屏，阶段性汇报即可）
3. **进度提示** - 每部分生成前显示当前进度（如"步骤 4/8 - 撰写 part_01/05"）
4. **附图说明必须生成 Mermaid 源码并在 docx 可见嵌入** - part_04 附图说明生成时，必须同时为每张图生成对应的 Mermaid 源码文件（.mmd），并在附图说明后以代码块形式可见嵌入该 Mermaid 源码（用于图片渲染异常时的兜底手工导入与调整）
5. **facts_ledger（强制）** - 分块撰写过程中必须持续落盘并更新 `artifacts/draft/facts_ledger.json`，用于跨块一致性与交付门禁：
   - 术语表（terminology）：术语/别名/定义必须统一
   - 图号登记（figure_registry）：每张图必须登记 image+mmd+editable（drawio/vsdx至少其一），并声明 `mermaid_source_embedded_in_docx=true`
   - 关键约束与效果（constraints_and_effects）：便于一致性审计/IPR审查定位
   - 要求：每生成/修改一个 part，必须同步更新 ledger；不得拖到最后一次性补齐
5. **不生成权利要求书** - 权利要求书由专利代理师撰写，不在交底书中提供
6. **选题合理性检查（强制，2026-03-09 新增）** - 撰写前必须检查应用场景的合理性和安全性
7. **内容差异化检查（强制，2026-03-09 新增）** - 与已有专利保持差异化，重复度≤30%
8. **发明名称字数强制校核（强制，2026-03-09 新增）** - 输出前必须校核字数≤25 字
9. **交付清理（可选）** - 默认保留分块 .md 与 versions 以便追溯与联动修改；若用户明确要求“交付洁净/只保留一个 docx”，再将过程性 .md 下沉到 `archive/` 或 `artifacts/`（不建议默认删除）
10. **创新示例验证（强制，2026-03-09 新增）** - 列举创新示例时必须先调研现有技术是否已实现，思考真实应用场景是否存在，不能编造伪创新
11. **技术组合策略（强制，2026-03-09 新增）** - 根据场景需求选择技术，智能体适用场景：多设备协同、多车协同、多任务协同；不适用场景：单车个性化、预测式配置

**字数约束（强制，2026-03-09 更新）：**
- 技术领域（part_01）：50-100 字
- 背景技术（part_02）：300-800 字
- 发明内容（part_03）：1200-2000 字（其中技术方案部分 800-1500 字）
- 附图说明（part_04）：100-300 字
- 具体实施方式（part_05）：1500-3000 字
- 发明名称：≤25 字（输出前必须校核）

**技术方案撰写要求（强制，2026-03-09 新增）：**
- 技术方案部分必须不少于 1500 字
- 必须详细描述每个技术特征的实现方式
- 必须包含具体的技术细节和实施例说明
- 必须说明各模块/步骤之间的连接关系和数据流
- 避免笼统描述，每个技术特征都要有具体实现支撑

**附图说明生成规范（2026-03-09 更新，2026-03-09 修正）：**

> 核心原则：**图片用于阅读交付；Mermaid 源码（.mmd）用于兜底复现与手工修图；可编辑图源（.drawio/.vsdx）用于可持续维护。**

- 每张图必须生成 Mermaid 源码文件（`.mmd`），并在**附图说明后以可见代码块**嵌入对应 Mermaid 源码（用于图片渲染异常时的兜底）。
- Mermaid 源码文件统一存放在项目的 `附图/` 目录（禁止再创建任何分散的附图子目录）。
- 文件命名建议：`fig_01_XXX.mmd`（与对应 png/svg、drawio/vsdx 同前缀，便于一键定位同一张图的全套素材）。
- **Mermaid 代码嵌入格式（强制）**：
  - 在附图说明中先给出“图 X 为……”的文字描述；紧接着给出该图的 Mermaid 源码块：
  ```markdown
  ## 四、附图说明

  图 1 为...；
  图 2 为...；

  **图 1 - 系统架构图（Mermaid 源码）：**
  ```mermaid
  graph TB
  A1[感知智能体模块]
  A2[场景识别智能体模块]
  A1 --> A2
  ```
  ```
- Mermaid 源码格式要求（ProcessOn 兼容，建议）：
  - 使用 `graph TD` 或 `graph LR` 开头（避免 `flowchart`）
  - 节点 ID 使用字母 + 数字组合（如 A1、B2）
  - 避免 `subgraph` / `style` 等易丢失特性
- **可编辑图源（强制）**：每张图至少保留一种可编辑图源（优先 `.drawio`，或 `.vsdx`）。
- **最终嵌图文件（强制）**：每张图必须导出最终嵌图图片（`.png` 或 `.svg`），并确保实际嵌入 docx。
- `.vsdx` 的生成时机：默认在最终交付阶段统一生成（避免迭代期反复导出）；若用户要求边写边出图，则允许在阶段内提前生成。

**背景技术撰写规范（强制，2026-03-09 新增）：**
- **必须引用至少 2-3 篇已有专利**
- **每篇专利必须包含**：专利号、公开日、申请人（如有）、技术内容概述、缺陷分析
- **引用格式**："中国发明专利 CNXXXXXXA（公开日 XXXX 年 XX 月 XX 日，申请人：XXX）公开了..."
- **缺陷分析转折**：用"然而""同时""此外"转折，不用"其缺陷在于""缺点是"
- **参考专利风格学习（强制，2026-03-10 新增）** - 当用户提供参考专利 PDF 或文本时，必须先使用 pdf skill 或 web_fetch 工具读取参考专利，学习其写作风格（如 CN121526509A 的"所述 XX 步骤包括..."格式），并在后续撰写中采用该风格

**具体实施方式撰写规范（强制，2026-03-10 更新）：**
- **实施例数量** - 只写 2 个实施例：一个方法实施例、一个系统实施例
- **叙述方式** - 使用"本发明实施例提供...""本发明实施例还提供..."的连贯段落，禁止使用"实施例一、实施例二"列表形式
- **方法实施例** - 结合流程图（图 2），按步骤顺序详细描述，每个步骤包含具体实现细节和示例
- **系统实施例** - 结合系统架构图（图 1），描述各模块的组成、连接关系和工作原理
- **示例嵌入** - 将具体示例（如"用户说'我累了'"）嵌入段落中，不用单独列出
- **结尾套话** - 必须以"以上所述，仅为本发明的具体实施方式，但本发明的保护范围并不局限于此..."结尾
- **风格要求** - 连贯段落叙述，避免 AI 腔调的列表式、结构化表达
- **写作风格（强制，2026-03-10 新增）** - 采用企业内部 IPR 审核的专利申请书格式，禁止使用"进一步"、"客户"、"贵方"等对外文档用语，改用"所述"、"本实施例"等专利规范用语
- **数据真实性（强制，2026-03-10 新增）** - 禁止编造实验数据、测试百分比、准确率等无实验支撑的数据（如"91% 准确率"、"35% 效率提升"），如需要数据必须标注"示例性数据，需实际测试"
- **实施例相关性（强制，2026-03-10 新增）** - 实施例必须与本发明技术领域直接相关，禁止复制参考专利中无关的实施例内容（如本发明是智能座舱，不得复制软件开发、项目管理等无关示例）

**进度提示格式：**
```
步骤 4/8 - 使用 modular-writer 分块撰写 [~180 秒]
正在撰写 part_01/05 - 技术领域 [~30 秒]
...
✅ part_01 完成，请确认是否继续 part_02？
```

**交付规范（强制，2026-03-10 更新）：**

1. **完整部分检查（强制）** - 生成 docx 前必须确认 5 个部分全部完成（part_01/02/03/04/05），任何部分缺失都不得生成 docx
2. **字数校验（强制）** - 每部分生成后必须检查字数，不达标必须重新生成或扩写
3. **文档路径展示** - 完成所有分块撰写后，必须展示完整文档路径列表
4. **合并后清理（强制）** - 生成.docx 后必须删除所有.md 中间文件（包括 part_*.md、合并版.md、附图说明.md 等）
5. **附图说明合并 mermaid 代码** - 合并版中的附图说明部分必须包含对应的 mermaid 代码
6. **Word 文档生成（强制）** - 必须生成 .docx 格式文件，使用 docx skill 或 docx-js 库
7. **格式校验（强制）** - 输出前必须确认交付物为 .docx 格式，不得交付 .md 格式
8. **文件结构整理（强制，2026-03-10 新增）** - 所有交付文件必须放在同一项目目录下：.docx 文件放在根目录；附图统一放在"附图/"子目录（包含 png/svg + mmd + drawio/vsdx）；禁止为“终稿/附图”另起 final/、drawings/ 等分散目录（统一“根目录 + 附图/”）
9. **风格一致性检查（强制，2026-03-10 新增）** - 输出前必须检查：无"进一步"等客户文档用语、无编造数据、实施例与本发明技术领域相关

## 工作流程

### 阶段 1：撰写准备

**步骤 1.1：确认撰写模式**

询问用户：
1. 撰写模式（专利/通用）
2. 文档主题
3. 是否有模板/风格参考

**步骤 1.2：加载分析结果（如有）**

检查是否存在：
- `workspace/analysis/template_rules.json` - 模板解析结果
- `workspace/analysis/style_profile.md` - 风格画像

如存在，加载并应用。

**步骤 1.3：确认分块结构**

**专利模式（默认）：**
```
1. 技术领域
2. 背景技术
3. 发明内容
4. 附图说明
5. 具体实施方式
```

**通用模式（用户自定义）：**
```
用户输入章节结构，如：
- 摘要
- 引言
- 相关工作
- 方法
- 实验
- 结论
```

---

### 阶段 2：分块撰写

**步骤 2.1：逐块生成**

按顺序生成每个部分：

```
生成 part_01 → 用户确认 → 生成 part_02 → 用户确认 → ...
```

**每块生成 Prompt 示例（专利模式）：**

```markdown
## 任务：撰写专利交底书的"技术领域"部分

**项目信息：**
- 专利名称：{{projectName}}
- 技术领域：{{techField}}
- 核心创新：{{coreInnovation}}

**写作规范（来自 template_rules.json）：**
- 目的：界定专利保护的技术边界
- 必须包含：所属技术领域、应用场景
- 结构：1-2 段连贯文字
- 避免：列表形式、过于宽泛的描述

**风格参考（来自 style_profile.md）：**
- 采用连贯段落叙述
- 先描述大领域，再聚焦到具体细分领域

**请生成：**
- 技术领域部分（200-300 字）
- 输出格式：Markdown
```

**步骤 2.2：落盘保存**

每生成一块，立即保存：

```python
# 伪代码
file_path = f"workspace/{project_name}/part_{index:02d}_{block_name}.md"
save_to_file(file_path, content)

# 记录元数据
update_metadata(block_name, status="completed", version=1)
```

**步骤 2.3：用户确认**

每块生成后询问用户：
```
✅ part_01_技术领域 已生成

内容摘要：
本发明涉及 XX 技术领域，尤其涉及一种...

[查看完整内容] [修改] [继续下一部分]
```

### 阶段 3：联动修改检测

**触发条件：**
- 用户修改某块内容
- 修改字数 > 100 字（可配置）

**检测流程：**

```
用户修改 part_05 → LLM 分析修改内容 → 识别依赖关系 → 
检测是否影响其他部分 → 输出联动修改建议
```

**LLM 分析 Prompt：**

```markdown
## 任务：分析修改内容的跨部分影响

**修改信息：**
- 修改文件：part_05_具体实施方式.md
- 修改内容：[新增/删除/修改的文本]

**请分析：**
1. 修改内容是否引入了新的技术特征？
2. 修改内容是否改变了技术方案？

### 阶段 5：IPR 模拟审查（专利模式专属）

**审查维度（9 项法定审查 + 2 项增强审查）：**

|------|---------|---------|------|
| **授权客体** | 是否属于技术方案 | 专利法第 2 条 | 通过/驳回 |
| **新颖性** | 是否属于现有技术 | 专利法第 22 条第 2 款 | 通过/警告 |
| **创造性** | 是否有实质性特点和进步 | 专利法第 22 条第 3 款 | 通过/警告 |
| **实用性** | 能否制造或使用 | 专利法第 22 条第 4 款 | 通过/驳回 |
| **说明书充分公开** | 是否清楚、完整、能够实现 | 专利法第 26 条第 3 款 | 通过/警告 |
| **权利要求得到支持** | 权利要求是否以说明书为依据 | 专利法第 26 条第 4 款 | 通过/警告 |
| **单一性** | 是否属于一个总的发明构思 | 专利法第 31 条 | 通过/警告 |
| **修改超范围** | 修改是否超出原范围 | 专利法第 33 条 | 通过/警告 |
| **诚实信用** | 是否违反诚实信用原则 | 专利法第 20 条 | 通过/驳回 |
| **形式审查**（增强） | 章节完整性、格式规范 | - | 通过/警告/错误 |
